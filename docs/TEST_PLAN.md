# Dialexis - План тестирования

## 1. Цель документа

Документ фиксирует стратегию тестирования проекта Dialexis, перечень ключевых тестовых сценариев и чеклист покрытия. Часть тестов должна быть реализована как основной костяк проекта, остальные фиксируются как обязательные или желательные сценарии для дальнейшего покрытия.

## 2. Цели тестирования

Тестирование направлено на подтверждение того, что:

- приложение корректно запускается в поддерживаемых режимах;
- peer-to-peer обмен сообщениями работает стабильно;
- сообщение передается с корректными полями;
- ошибки пользовательского ввода и подключения обрабатываются предсказуемо;
- ключевые части логики остаются проверяемыми при дальнейших изменениях.

## 3. Область тестирования

В тестовый контур входят:

- разбор и валидация CLI-аргументов;
- доменная модель сообщения;
- прикладная логика отправки и приема сообщений;
- преобразование между доменной моделью и protobuf-моделью;
- gRPC-транспорт между двумя peer-узлами;
- базовые пользовательские сценарии запуска.

Вне базового контура:

- нагрузочное тестирование;
- тестирование GUI;
- тестирование шифрования и аутентификации, так как они не входят в требования задания.

## 4. Подход к тестированию

Для проекта используется несколько уровней тестов:

- модульные тесты для изолированной проверки классов и преобразований;
- интеграционные тесты для проверки обмена между двумя peer-узлами;
- ручное сценарное тестирование для консольного интерфейса и поведения приложения при запуске.

Основной автоматизированный костяк должен покрывать критический путь:

- корректный разбор аргументов;
- корректное создание и отображение сообщения;
- корректное преобразование `ChatMessage <-> protobuf`;
- успешную доставку сообщения между двумя peer-узлами;
- обработку базовой ошибки подключения.

## 5. Тестовые уровни

### 5.1. Модульное тестирование

Цель:

- проверить отдельные классы без сетевой инфраструктуры;
- быстро выявлять регрессии в прикладной логике.

Основные кандидаты:

- `CliOptions`;
- `ChatService`;
- `ProtoMessageMapper`;
- форматирование сообщений для вывода.

### 5.2. Интеграционное тестирование

Цель:

- проверить взаимодействие gRPC-клиента и gRPC-сервера;
- убедиться, что сообщение доходит от одного узла до другого с сохранением данных.

Основные кандидаты:

- `GrpcChatClient + ChatGrpcEndpoint + GrpcChatServer`;
- `PeerSessionManager` в сценарии обмена сообщениями.

### 5.3. Ручное тестирование

Цель:

- проверить реальные пользовательские сценарии запуска;
- убедиться, что консольный интерфейс удобен и не ломается при входящих сообщениях;
- проверить понятность сообщений об ошибках.

## 6. Основные риски

Наиболее важные риски, которые должны быть покрыты тестами:

- приложение запускается с невалидными аргументами;
- сообщение отправляется без обязательных полей;
- теряются поля сообщения при преобразовании в protobuf;
- соединение с peer не устанавливается или рвется;
- входящее сообщение не попадает в прикладной слой;
- приложение некорректно ведет себя при запуске без удаленного peer.

## 8. Основной костяк тестов для реализации

Для сдачи проекта в первую очередь рекомендуется реализовать следующие тесты как минимально достаточный автоматизированный набор:

- `UT-01` Разбор валидных CLI-аргументов;
- `UT-02` Ошибка при отсутствии обязательного `--name`;
- `UT-05` Преобразование `ChatMessage` в protobuf;
- `UT-06` Преобразование protobuf в `ChatMessage`;
- `UT-07` `ChatService.sendMessage()` создает корректное сообщение;
- `UT-08` `ChatService.onIncomingMessage()` уведомляет listener/UI;
- `IT-03` Доставка одного сообщения между двумя peer-узлами;
- `IT-05` Ошибка подключения к недоступному peer.

Этот набор покрывает критический путь: старт приложения, корректность данных сообщения, прикладную логику и сетевую доставку.

## 9. Чеклист тестов

### 9.1. Чеклист автоматизированных тестов

- [ ] Реализован тест на корректный разбор CLI-аргументов.
- [ ] Реализован тест на отказ запуска без обязательного имени пользователя.
- [ ] Реализован тест на отказ запуска без обязательного локального порта.
- [ ] Реализован тест на согласованность `connect-host/connect-port`.
- [ ] Реализован тест на преобразование `ChatMessage -> protobuf`.
- [ ] Реализован тест на преобразование `protobuf -> ChatMessage`.
- [ ] Реализован тест на создание сообщения в `ChatService`.
- [ ] Реализован тест на обработку входящего сообщения в `ChatService`.
- [ ] Реализован интеграционный тест на доставку сообщения между двумя peer-узлами.
- [ ] Реализован интеграционный тест на ошибку подключения.

### 9.2. Чеклист ручной проверки

- [ ] Узел запускается в режиме ожидания входящего подключения.
- [ ] Узел подключается к уже запущенному peer.
- [ ] Сообщение отображает имя отправителя.
- [ ] Сообщение отображает дату и время отправки.
- [ ] Сообщение отображает текст без потери символов.
- [ ] Входящие сообщения видны в консоли без перезапуска приложения.
- [ ] Ошибка недоступного peer отображается понятным образом.
- [ ] Приложение корректно завершается после остановки.

## 10. Инструменты тестирования

Предлагаемые инструменты:

- `JUnit 5` для модульных и интеграционных тестов;
- `Mockito` при необходимости изолировать listener или сервисные зависимости;
- встроенные средства gRPC и локальные порты для интеграционных тестов;
- ручной запуск двух экземпляров приложения в отдельных терминалах.

## 11. Критерии завершения тестирования

Тестирование можно считать достаточным для базовой сдачи, если:

- реализован основной костяк автоматизированных тестов;
- пройдены ключевые ручные сценарии;
- нет критических дефектов в запуске, доставке и отображении сообщений;
- документирован перечень оставшихся, но запланированных тестов.

## 12. Вывод

Данный тест-план можно использовать как артефакт для сдачи и как рабочий ориентир для команды. Он отделяет обязательный тестовый минимум от полного желаемого покрытия и позволяет показать, что проект тестируется осознанно, а не только точечно по ходу разработки.
