# Dialexis - План тестирования

## Цели тестирования

Тестирование направлено на подтверждение того, что:

- приложение корректно запускается в поддерживаемых режимах;
- peer-to-peer обмен сообщениями работает стабильно;
- сообщение передается с корректными полями;
- handshake позволяет обеим сторонам инициировать отправку после подключения;
- ошибки пользовательского ввода и подключения обрабатываются предсказуемо;
- ключевые части логики остаются проверяемыми при дальнейших изменениях.

## Область тестирования

В тестовый контур входят:

- разбор и валидация CLI-аргументов;
- доменная модель сообщения;
- прикладная логика отправки и приема сообщений;
- преобразование между доменной моделью и protobuf-моделью;
- gRPC-транспорт между двумя peer-узлами;
- handshake-регистрация peer;
- базовые пользовательские сценарии запуска;
- форматирование сообщений и системных статусов.

## Подход к тестированию

Для проекта используется несколько уровней тестов:

- модульные тесты для изолированной проверки классов и преобразований;
- интеграционные тесты для проверки обмена между двумя peer-узлами;
- ручное сценарное тестирование для консольного интерфейса и GUI.

Основной автоматизированный костяк покрывает критический путь:

- корректный разбор аргументов;
- корректное создание и отображение сообщения;
- корректное преобразование `ChatMessage <-> protobuf`;
- корректную регистрацию peer через handshake;
- успешную доставку сообщения между двумя peer-узлами;
- двусторонний обмен сообщениями;
- обработку ошибки подключения.

- `JUnit 5` для модульных и интеграционных тестов;

- ручной запуск двух экземпляров приложения в отдельных терминалах;
- ручной запуск JavaFX GUI через `./gradlew runGui`.
